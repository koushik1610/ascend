# Number & Honesty Policy (binding, every file)

The system's credibility depends on this. Two ideas: **be honest** and **be discreet**.

## Honesty gates (absolute — never cross)
- **Never fabricate** experience, job titles, dates, metrics, certifications, skills, or referral
  contacts. If the user can't substantiate it under follow-up questioning, it does not go on a resume,
  a signal page, or in a sendable message.
- **A missing number is not a license to invent one.** Suggest *what the user could measure* to get a
  real number; leave the bullet qualitative until they have it.
- **Expired/in-progress credentials** are labeled as such — never shown as active.
- **No claimed expertise the user lacks.** Where a target role wants something the user doesn't have,
  that's a *gap with honest handling* (a learning plan, an adjacent-strength pivot, or an honest "I'd
  ramp on that here"), never a bluff.
- **Personal-conviction writing** (a "why this company/mission" essay, a values answer) is **never
  generated as finished prose** — the system produces honest *beats/outlines*; the user writes it in
  their own voice. Generated conviction reads fake and is often screened for.
- **Referral names are real or absent.** Seed referral suggestions only from the user's actual network
  (the LinkedIn export). Never invent a contact or a relationship.

## Number discipline (two tiers)
Each number a user can claim falls into one of two buckets:

1. **Own / verifiable numbers — stay EXACT, everywhere.** Numbers the user personally produced or that
   are externally verifiable: benchmark results, public metrics, project counts, years of experience,
   portfolio stats. Use the precise figure; precision here = credibility. (Scope a result to its
   context — e.g., a score "on the X benchmark," a lift "for the Y cohort" — so it can't read as an
   unbounded claim.)

2. **Employer-internal operational numbers — SANITIZE on anything sendable.** Numbers that describe an
   employer's internal scale or that the user shouldn't publish (exact headcounts, exact account/seat/
   customer counts, internal revenue, unreleased metrics, internal project codenames). On any surface
   that leaves the machine — resume, signal, LinkedIn, a DM — **round them or remove them**, and never
   name internal codenames. The exact figures may live in *private* prep notes (clearly marked
   internal) and may be spoken in interviews where appropriate, at the user's discretion.

> Precision about an employer's internals reads as a confidentiality risk to an interviewer.
> "I'll keep the exact internal split vague" is a *positive* senior signal, not a weakness.

### Recording dual-value numbers
In the master resume's metrics bank, store sanitized numbers as pairs:
`[INTERNAL: <exact> → PUBLIC: "<rounded/sanitized>"]`. Every derived resume and signal uses ONLY the
public value. The user sets the specific rounding rule during intake (Phase 0) — capture it in
`intake.md` and apply it consistently.

## The pre-delivery check
Before declaring any sendable artifact done, grep it for the user's never-publish list (their exact
internal figures and codenames from `intake.md`). Zero hits required. Confirm every remaining number
is either an own/verifiable exact figure or an approved public rounding.

## Default rule (if the user has no sensitive data)
Many users (especially outside large enterprises) have nothing to sanitize. Then: keep numbers exact,
still never fabricate, still label credentials honestly, still no invented referrals. The honesty
gates always apply; the sanitization tier only applies when the user flags sensitive internals.
